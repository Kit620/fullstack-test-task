## Тестовое задание на позицию Fullstack разработчика (Python + React)

**Вводные:**
1. Здесь представлен MVP проект файлообменника. Он позволяет загружать файлы, проверяет их на подозрительный контент и отправляет алерты;
2. Репозиторий содержит в себе бэкенд и фронтенд части;
3. В обоих частях присутствуют баги, неоптимизированный код, неудачные архитектурные решения.

**Задачи:**
1. Проведите рефакторинг бэкенда, не ломая бизнес-логики: предложите свое видение архитектуры и реализуйте его;
2. (Дополнительно) На бэкенде есть возможность неочевидной оптимизации - выполните ее;
3. (Дополнительно) Разбейте логику фронтенда на слои;

**Запуск:**
1. ```docker compose -f docker-compose.dev.yml up```
2. ```docker exec -it backend alembic upgrade head```


**Открыть фронт:** ```http://localhost:3000/test``` 

**Открыть бэк:** ```http://localhost:8000/docs```

---

**Что реализовано:**

- Рефакторинг бэкенда: слои "Router -> Service -> Repository" вместо одного файла с бизнес-логикой и прямыми обращениями к БД. Подключение к БД вынесено в единый `backend/src/db.py`, бизнес-правила - в `backend/src/services/`, SQL-запросы - в `backend/src/repositories/`.
- Оптимизация бэкенда: обработка загруженного файла ("проверка на угрозы -> извлечение метаданных -> создание алерта") объединена в одну Celery-задачу с одной транзакцией вместо трёх последовательных задач с тремя отдельными сессиями и коммитами.
- Разбивка фронтенда на слои: "Component -> Hook -> API-клиент" вместо одного файла `page.tsx` - `frontend/src/types/`, `frontend/src/api/`, `frontend/src/hooks/`, `frontend/src/components/`.
- Дополнительно: баг `DELETE /files/{id}` (падал с 500, если у файла уже были алерты - не хватало `ON DELETE CASCADE` на внешнем ключе), добавлена кнопка удаления файла в UI (раньше удаление было доступно только напрямую через API), подключены и настроены `ruff`/`mypy` на бэкенде и `eslint`/`prettier`/`vitest` на фронтенде.

**Откуда заходить:**

- Фронтенд (список файлов, загрузка, алерты) - `http://localhost:3000/test`
- Backend Swagger (интерактивная документация и ручные вызовы API) - `http://localhost:8000/docs`
- БД напрямую - `docker exec -it backend-db psql -U postgres -d test` (или любой GUI-клиент на `localhost:5433`, база `test`, логин/пароль `postgres`/`postgres`)

**Как посмотреть, что реально работает:**

1. Поднять стек и применить миграции (см. "Запуск" выше).
2. Загрузить файл через UI или `curl -X POST http://localhost:8000/files -F "title=test" -F "file=@/path/to/file"` - статус обработки и алерт появятся через пару секунд, пока файл обрабатывается в фоне через Celery.
3. `docker compose -f docker-compose.dev.yml logs backend-worker` - в логе будет ровно одна пара `received`/`succeeded` на файл (результат объединения трёх задач в одну, см. п.2 выше).
4. Удалить файл кнопкой в таблице - запись и связанный с ней алерт исчезнут (проверка каскадного удаления).
